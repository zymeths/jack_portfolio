Building a Loan Default Predictor: From Messy Notebook to 0.924 AUC
What I learned going from messy raw financial data to a deployable loan default classifier — preprocessing decisions, model selection, and the trade-offs nobody writes about.
[ + read ]
May 2026 machine learning 10 min read

I recently competed in Kaggle's Playground Series S5E11 — a binary classification problem predicting whether a borrower would pay back their loan. The dataset had 600k rows, an 80/20 class imbalance, and was evaluated on AUC-ROC. This is a writeup of every meaningful decision I made, including the ones that backfired.

The Data

Eleven features: a mix of financial metrics (debt_to_income_ratio, credit_score, interest_rate, loan_amount, annual_income) and demographic categoricals (employment_status, loan_purpose, grade_subgrade, education_level, gender, marital_status).

The first thing I confirmed was the class split: 79.9% paid back, 20.1% default. A 4:1 imbalance isn't catastrophic, but it's significant enough that accuracy is a completely useless metric. From the very first cell, I tracked AUC-ROC only. No missing values — unusual for real lending data, consistent with the synthetic generation process this competition used.

EDA Findings That Actually Mattered

debt_to_income_ratio dominates. When I ran feature importances after the first model, this feature was 2.5x more important than the second-placed feature. That gap held up throughout tuning.

grade_subgrade has a clean ordinal signal. Plotting default rate by grade showed a near-perfect gradient — A grades default least, F grades default most. This told me immediately that label encoding was the wrong choice here. It needed ordinal encoding.

interest_rate underperformed expectations. In real-world lending models, interest rate is typically a top predictor. Here it ranked below employment_status — a quirk of synthetic data generation worth noting but not worth chasing.

gender and marital_status contributed almost nothing. Feature importances confirmed what the EDA hinted at. They got dropped before the final model.

Modelling Approach

I started with three models at default parameters to establish a floor: Logistic Regression, LightGBM, and XGBoost. Logistic Regression scored 0.8725 AUC. Both tree models scored 0.9201 — identical to four decimal places, which told me the ceiling at default parameters was data-determined, not model-determined.

That jump from 0.87 to 0.92 confirmed tree models were the right direction. The question was how much I could push past 0.92.

Encoding matters more than I expected. My first pass used label encoding on all categoricals — fast, simple, a reasonable starting point. I moved to ordinal encoding for grade_subgrade and education_level, and target encoding for loan_purpose and employment_status.

Feature engineering. I created four ratio features grounded in lending logic: loan-to-income, interest burden, implied total debt, and loan-to-credit score. I also tried credit score bands, interest rate bands, and splitting grade_subgrade into components — feature importances revealed all of these were near-zero. LightGBM already handles binning internally, so manually binning just lost information.

Hyperparameter tuning with Optuna. Rather than grid search, I used Bayesian optimisation. 50 trials pushed the CV score from 0.9201 to 0.9231.

The Leakage Problem

This is the most important part of the writeup.

I submitted my first model and got 0.80 AUC on the leaderboard — a 12-point gap from my 0.92 CV score. That gap is the signature of data leakage.

The culprit was target encoding. I calculated encoded values using the entire training set including its target, then trained and validated on that same data. Every validation row had already leaked its target information into its own features. The CV score looked great because the model was essentially peeking at the answers. On Kaggle's test set there was no leakage, and the true performance was exposed.

The fix was to move target encoding inside each cross-validation fold: fit the encoder on fold training rows only, transform validation and test rows using that fold's mapping, never let a validation row influence its own encoding.

I also switched from standard KFold to Stratified KFold, which guarantees each fold maintains the same 80/20 class ratio. And rather than fitting one model and predicting once, I averaged test predictions across all 5 folds — a simple form of ensembling that reduces variance.

The result: leaderboard score jumped to 0.924. The honest version of this story is that 0.924 was always the true performance. The 0.92 CV score with leakage wasn't a good model — it was a misleading measurement.

What the Final Pipeline Looks Like

The final model is a LightGBM classifier with Optuna-tuned hyperparameters, trained inside a 5-fold stratified loop with leakage-free target encoding. The codebase is split into three files:

FilePurpose
features.pyAll encoding and feature engineering logic as importable functions
train.pyFull training pipeline, runnable as a single command
evaluate.pyROC curves, feature importances, prediction distributions

The entire pipeline reproduces the 0.924 submission with python src/train.py.

What I'd Do Differently

Tune target encoding smoothing as a hyperparameter. I fixed smoothing=10 throughout. It controls how aggressively rare categories are pulled toward the global mean — worth tuning alongside model parameters.

Check train/test distribution shift earlier. I planned to compare feature distributions between train and test in the EDA and skipped it. That check would have flagged the leakage risk sooner.

Blend LightGBM and XGBoost properly. I experimented with blending but the final submission used LightGBM only. With leakage-free OOF predictions from both models, a weighted blend often gives another small lift because the two models make different errors.

The Lesson That Sticks

Cross-validation score is only a meaningful number if it accurately simulates what will happen on unseen data. The moment any information from a validation row touches the encoding, features, or model that predicts it, the CV score becomes optimistic and untrustworthy.

This is why production ML pipelines treat preprocessing as part of the model — not a separate step that happens beforehand. Sklearn's Pipeline object exists precisely to enforce this boundary. It's the next thing I'm building into this project.

RFM Segmentation: When Simple Models Beat Complex Ones
Why a well-structured RFM analysis outperformed clustering models for our use case, and how we packaged the output so product teams could actually act on it.
[ + read ]
May 2026 analytics 7 min read

There's a temptation in data science to reach for the most sophisticated tool available. When our team was asked to segment coupon users for a super-app platform, the instinct was to run clustering — K-Means, DBSCAN, maybe something hierarchical. Let the algorithm find the natural groupings.

We didn't. We built an RFM model instead. And it was the right call.

The Problem

We had coupon transaction data spanning two years and tens of thousands of users. The business question was simple: who are our users, and how should we treat them differently?

The "obvious" answer was clustering. But before reaching for sklearn, we asked a more important question: what does the product team actually need to do with this output?

The answer was: target users with different coupon strategies. Re-engage lapsing users. Reward loyal ones. Push high-value offers to high-spend segments. That's a retention and CRM problem. And for that, RFM is almost always the better fit.

Why RFM, Not Clustering

Clustering finds natural groupings in your data — but those groupings aren't always actionable. You might end up with a cluster defined by "users who buy on Tuesdays and prefer cosmetics," which is interesting but hard to build a campaign around.

RFM gives you segments grounded in three dimensions that every marketer and product manager already understands: Recency — how recently did they buy? Frequency — how often? Monetary Value — how much do they spend? These map directly to business levers. You know exactly what to do with a high-R, low-F user versus a low-R, high-M user. The model is interpretable by construction.

The Scoring Approach

We scored each dimension on a 1–5 scale. For Recency, the bands were time-based. For Frequency and Monetary Value, we used percentile thresholds.

ScoreRecencyFrequency / Monetary
5Within 30 daysTop 5%
4Last 3 monthsTop 20%
3Last 6 monthsTop 30%
2Last yearTop 60%
1Over a year agoRest

This produces a three-digit RFM score per user (e.g. 554, 312, 111), giving 125 possible combinations. Rather than treating each combination separately, we collapsed them into 11 named segments — which is where the real value came from.

The Segments

Instead of "Cluster 3" or "High-R Low-F," we gave every segment a name that made the business logic self-evident.

SegmentRFM ProfileWhat it means
Soul Mate555Best users — recent, frequent, high spend
Lover553, 534, 544…Loyal, consistently high RFM
New Passion514, 515, 525…New but spending big
Flirting414, 415, 424…Recent, high value, but infrequent
Potential332, 333, 353…Solid middle-of-the-road users
About to Dump You212, 221, 322…Disengaging — critical at-risk group
Ex113, 133, 153…Used to be great customers
Break-up (Lost)111, 112, 122…Gone

The naming wasn't just fun — it made the segments sticky. When a PM says "let's focus on the About to Dump You group this sprint," everyone in the room immediately knows what that means and why it matters.

What the Data Showed

The "About to Dump You" segment was the largest at-risk group — over 3,500 users — but their average monetary value was still decent. These weren't low-value users walking away. They were disengaging despite having spent meaningfully before. That's a winnable segment if you act fast.

The Apprentice segment was large but misunderstood. Nearly 5,000 users with very recent activity but low frequency and spend. Easy to write off — but we flagged them as a growth opportunity. They were new and active; they just hadn't found the right coupon category yet.

Soul Mates (555) were only 56 users, but averaged 43+ purchases and a monetay mean of ~947k. The recommendation was simple: don't neglect them. Loyalty rewards, exclusive early access, anything that signals they're seen.

Year-over-year, the overall user base grew but quality eroded — the Apprentice and About-to-Dump segments had grown fastest, suggesting the platform was good at acquisition but struggling with retention.

The Recommendations

The segmentation was only useful if it drove action. We packaged the output into four targeted strategies:

1. Win back the "About to Dump You" group with one-time-use, low-friction coupons. Our data showed this segment over-indexed on affordable, habitual purchases — coffee, pizza, casual dining. A cheap, easy win is the right re-engagement hook.

2. Nurture Apprentices while recency is still high. They haven't disengaged yet. Serve them low-cost, high-volume coupon categories to build the habit before they drift.

3. Reward the top segments. Soul Mates, Lovers, and New Passions shouldn't receive the same campaigns as everyone else. Loyalty gifts or exclusive coupons reinforce the behaviour you want to keep.

4. Use weaker segments for research. Offering a small discount ("fill out this survey, get 4% off your next coupon") on lost or Ex users is a low-cost way to understand churn without a separate research budget.

What I'd Do Differently

RFM has real limitations worth being honest about. The model is static — a user's segment doesn't update until you re-run the scoring. In a live product, you'd want this running on a scheduled pipeline, not a one-off analysis.

The monetary thresholds were also percentile-based, which means a "top 5%" user in a thin dataset might not be high-value in absolute terms. Category-level RFM — separate scoring for food coupons vs. product coupons — would have been more nuanced, but the added complexity wasn't worth it for a first iteration.

A well-scoped simple model that gets used beats a sophisticated model that gets shelved. We could have spent weeks on clustering and presented something that impressed in a demo but left the product team with no clear next step. Instead, they had 11 named segments and four campaign strategies by end of sprint.
